# AI CG Studio · 个人设计手册

> 状态：持续维护
>
> 定位：个人创作工具，自用为主，偶尔分享给身边朋友
>
> 核心角色：绫地宁宁、四季夏目

## 1. 为什么做这个工具

实际创作中最费时间的往往不是点击“生成”，而是从空白 Prompt 开始决定：画谁、发生了什么、用什么镜头、什么光线，以及怎样让角色保持一致。

AI CG Studio 把这些决定整理成可复用的 Scene。打开网站后先选一个想画的瞬间，再调整故事和画面，最后由工具组合 Prompt 并调用 SD WebUI。

这个项目不承担“服务所有 AI 创作者”的目标。判断一个功能是否值得做，只看它能否让自己和朋友更快、更稳定地完成一张想要的 CG。

## 2. 使用范围

### 主要使用者

- 我自己：整理灵感、测试角色 LoRA、保存稳定参数和生成记录
- 身边朋友：通过临时链接选择场景并出图

### 不是这个项目的目标

- 公开运营的在线绘图服务
- 用户账号、社交关系和社区投稿
- Scene 商店、订阅或付费体系
- 长期公网托管和云端数据同步
- 为所有模型、角色和工作流提供通用解决方案

保持边界可以减少维护成本，也能让本地数据、SD 连接和场景质量得到更多注意。

## 3. 核心创作流程

```text
灵感
  ↓
故事：这一刻发生了什么
  ↓
Scene：角色、情绪、镜头、构图、光照、色彩
  ↓
Prompt：把 Scene 翻译成模型能理解的标签
  ↓
Image：调用 SD WebUI 生成
  ↓
Review：评分、收藏、备注、重新生成或做变体
```

Prompt 仍然可以查看、复制和修改，但不应成为打开网站后必须面对的第一件事。

## 4. 设计原则

### 4.1 从 Scene 开始

场景不是“教室”“海边”这样的地点词，而是一个具体瞬间，例如“雨停后的共伞余温”或“放学后没有说出口的等待”。标题负责唤起画面，故事负责说明人物关系，Prompt 负责落地。

### 4.2 角色一致性优先

当前内容围绕绫地宁宁和四季夏目展开。发色、瞳色、发型、服装、气质和 LoRA 触发词需要保持一致。背景、特效和复杂构图不能抢走角色主体。

### 4.3 情绪要能被看见

“害羞”“安心”“期待”不能只停留在分类标签里，还要通过视线、动作、距离、光线和环境反馈出现在画面中。

### 4.4 内容质量优先于数量

新增场景前先检查已有场景是否重复。一个可用的 Scene 应该有清楚的故事、合理的构图、准确的角色提示词和可复现的参数，而不是只增加一个标题。

### 4.5 本地和可恢复优先

项目默认在本机运行。个人历史、收藏和图片保存在浏览器本地；重要内容应能下载、导出或从 JSON 恢复。任何分享能力都应可以随时停止。

## 5. 信息架构

| 页面 | 主要任务 | 不应承担的任务 |
|---|---|---|
| 首页 | 快速开始、继续最近创作、进入场景库 | 展示大量技术参数 |
| 场景库 | 搜索、筛选、比较 Scene | 直接塞满完整 Prompt |
| 导演工作台 | 调整故事与画面、生成、保存 | 让使用者手动拼接所有标签 |
| 角色页 | 查看角色设定、LoRA 与推荐场景 | 重复维护一套独立场景标题 |
| 风格页 | 选择色彩、光照和氛围 | 覆盖角色与故事设定 |
| 画廊 | 回看图片、参数、评分和收藏 | 伪装成公开作品社区 |
| 知识库 | 记录创作方法和维护约定 | 写成商业产品宣传页 |

## 6. Scene 内容规范

### 6.1 标题

- 优先使用能够想象出画面的中文短句
- 尽量包含时间、动作、关系或情绪中的至少一项
- 避免只写地点、服装或抽象概念
- 角色页、首页和推荐区统一读取 Scene 标题，不另外手写旧标题

### 6.2 故事

一个故事至少应回答：

1. 谁在场？
2. 发生在哪里、什么时候？
3. 正在发生什么动作或互动？
4. 这一刻为什么值得画？

故事以能指导画面为准，不需要写成长篇小说。宁宁与夏目的语气、距离感和行为方式应有区分，避免把名字替换后仍然完全相同。

### 6.3 Prompt

Prompt 需要与故事中的人数、动作、服装、场所、季节和光照一致。建议按质量、人数、角色、服装、场景、动作表情、镜头构图、光照氛围、LoRA 的顺序组织；具体顺序见 `docs/prompt-spec.html`。

推荐参数是起点，不是不可修改的规则。模型、VAE、LoRA 版本或显存条件变化后，应以实际测试结果为准。

### 6.4 推荐场景

角色页和其他推荐区只保存 Scene ID，显示时从 `data/scenes.json` 读取当前标题与故事。这样修改场景文案后不会留下旧标题。

## 7. 文案规则

- 语气自然、直接，像给自己和朋友写说明
- 中文优先；保留 Scene、Prompt、LoRA、SD WebUI 等确实需要的术语
- 不使用“重新定义行业”“服务所有创作者”“官方商店”等超出项目范围的表达
- 不用虚假的精确承诺，例如“95% 稳定”“一次即可出图”
- 功能状态以网站当前实际能力为准，未完成内容明确写成“可选”或“以后有需要再做”
- 按钮、README、启动指南和页面提示使用同一名称
- 错误提示要告诉使用者下一步能检查什么

## 8. SD WebUI 与分享

### 本地连接

- Stability Matrix 中的 WebUI 使用 `--api`
- 控制面板允许填写实际本机地址，不假定端口一定是 7860
- 工作台通过本地网关访问 `/sdapi`，普通静态服务器只负责浏览
- 默认使用 WebUI 当前模型，也允许单次选择其他模型和采样配置

### 朋友分享

- 分享链接使用临时公网通道和随机 Token
- 本机直连不要求 Token，公网请求必须验证
- 停止分享后，临时入口应立即失效
- Token 只发给信任的人，不把网站描述成可公开部署的服务
- 多人共用同一个 WebUI 时，要明确“停止生成”会中断当前全局任务

### 状态和错误

至少区分：

- 当前只是静态页面，未启用 SD 网关
- 网关已启动，但 WebUI 未响应
- WebUI 已连接，可以读取模型和参数
- 公网通道尚未建立或本机没有安装 `cloudflared`

## 9. 数据与隐私

- 仓库 JSON 保存默认场景、角色、标签和参数
- IndexedDB 保存当前浏览器中的历史、图片、项目与收藏
- 项目不提供账号和云端数据库
- 清理浏览器数据可能丢失本地记录，重要作品需要下载或另行备份
- 分享模式会把生成请求转发到本机 SD WebUI，不应向陌生人公开链接

## 10. 质量检查

### 场景发布前

- 标题与故事是否具体、自然、没有重复
- 角色设定与 LoRA 是否匹配
- Prompt 是否与人数、动作、季节和场所一致
- 是否包含容易冲突或互相抵消的标签
- 推荐参数是否只是参考值，没有写成绝对保证
- 首页、角色页和推荐区是否显示同一个标题
- `npm run validate` 是否通过

### 出图后

按人物、表情、构图、手部和氛围五项记录结果。低分不代表场景必须删除，而是帮助判断需要改故事、Prompt、参数还是 LoRA。

## 11. 维护优先级

### 当前优先

1. 优化宁宁与夏目的旧场景故事和 Prompt
2. 保持角色页、首页和场景库文案一致
3. 改善 SD 连接诊断、错误提示和分享稳定性
4. 完善历史、收藏、项目和本地备份体验
5. 保持桌面与手机端的基本可用性

### 有实际需要再做

- 更方便的导入、导出和备份
- 批量对比同一 Scene 的不同参数
- 服装与 LoRA 预设解耦
- ComfyUI 工作流导出
- ControlNet、IP-Adapter 或参考图辅助
- 小范围的本地 AI 文案与构图建议

### 暂不计划

- 公开账号体系
- 社区投稿与审核
- Scene 商店或订阅
- 长期托管的云端服务

更具体的状态见 `docs/roadmap.html`。路线会随实际使用调整，不为了版本号赶工。

## 12. 做决定时先问

1. 这个改动会让自己或朋友更容易完成一张 CG 吗？
2. 它会增加多少启动、维护或排错成本？
3. 是否能保持本地优先和随时停止分享？
4. 是否会让角色、场景和 Prompt 的数据出现重复来源？
5. 如果半年不维护，它还能继续工作吗？

只有答案仍然合理时，功能才值得进入项目。

---

> Prompt 描述图片，Scene 描述瞬间。设计的目标不是把工具写得宏大，而是让想画的那个瞬间更容易落地。
